Cómo está montado

La idea de fondo es simple: separar responsabilidades. El modelo (las reglas) no sabe nada de la consola, y la interfaz no sabe nada de las reglas. Así se puede cambiar el aspecto sin tocar la lógica, y probar la lógica sin necesidad de la interfaz.

◆ Estructura de paquetes

src/com/tuio/juegoxturnos/
├── Main.java              # arranque: crea dependencias y lanza el juego
├── combate/               # orquestación del combate
│   ├── Juego                # modos de juego y menús
│   ├── Combate              # motor por turnos (simétrico)
│   ├── Combatiente          # personaje + inventario + control + nombre
│   ├── Controlador          # interfaz: quién decide el turno
│   ├── ControladorHumano    # pide la acción por consola
│   ├── ControladorCpu       # delega en la IA
│   └── AccionTurno          # sealed: Atacar | Defender | UsarObjeto
├── ia/
│   └── EstrategiaCPU        # decisiones de la máquina
├── modelo/               # las reglas del juego (sin I/O)
│   ├── Personaje, Ataque, ResultadoAtaque
│   ├── personajes/          # Guerrero, Mago, Arquero, Asesino
│   ├── efectos/             # EfectoEstado, DanioPorTurno, Aturdimiento…
│   └── items/               # Item, PocionVida, Escudo, Antidoto, Inventario
├── ui/
│   ├── Consola              # marcos, barras, menús, narración
│   ├── ArteAscii            # dibujo de cada personaje
│   └── Colores              # paleta ANSI
└── util/
    └── Aleatorio            # aleatoriedad centralizada (con semilla)

◆ Las capas

modelo

El corazón del juego: personajes, ataques, efectos de estado e ítems. No imprime nada ni lee teclado; solo aplica reglas. Muchos objetos son inmutables (Ataque, ResultadoAtaque).

combate

Orquesta el duelo. Combate lleva el turno de cada bando; Juego gestiona los modos y menús; AccionTurno representa lo que se hace en un turno.

ia

EstrategiaCPU decide: curarse si está malherida, usar antídoto si está envenenada, defenderse si peligra o atacar (rematando con el especial cuando conviene).

ui

Todo el dibujo vive aquí: Consola pinta marcos, barras y el registro de sucesos; ArteAscii guarda los dibujos; Colores centraliza la paleta.

util

Aleatorio encapsula el azar. Se puede fijar una semilla para partidas reproducibles y para los tests.

Main

Solo cablea las piezas (consola, aleatoriedad) y arranca Juego. Sin lógica de juego.

◆ Patrones y decisiones

  • Estrategia (Controlador): el motor de combate no distingue humano de CPU. Cada bando tiene un Controlador que decide su turno. Por eso el mismo motor sirve para 1 jugador, 2 jugadores y torneo.
  • Tipos sellados (sealed): AccionTurno solo puede ser Atacar, Defender o UsarObjeto. El compilador ayuda a no olvidar ningún caso.
  • Inmutabilidad: los resultados y los ataques son record sin estado mutable; menos errores y más fáciles de razonar.
  • Fábricas: los efectos (Efectos.veneno()), el inventario (Inventario.porDefecto()) y los personajes (vía Supplier) se crean en un único sitio, fácil de ajustar para el balance.
  • Aleatoriedad inyectada: nadie crea su propio Random; todos reciben el mismo Aleatorio. Eso hace el juego testeable.
  • Separación de I/O: la lógica devuelve datos (p. ej. ResultadoAtaque) y la Consola los narra. La lógica no imprime.

◆ El flujo de un turno

Cada turno de un combatiente sigue estos pasos:

1. Se dibuja el escenario (ambos personajes, vida, maná y sucesos).
2. Se levanta la defensa del turno anterior.
3. Se procesan los efectos de estado (veneno/quemadura/sangrado hacen daño).
   → si el efecto lo mata, termina el combate.
   → si está aturdido, pierde el turno.
4. El Controlador decide la acción: Atacar, Defender o UsarObjeto.
5. Se ejecuta y se narra el resultado (daño, crítico, esquiva, efecto…).
6. El combatiente regenera algo de maná.
Al atacar, primero se comprueba si el objetivo esquiva; si no, se calcula el daño (con posible crítico ×1.5), se aplica sobre el escudo y la vida, y —si el ataque lo lleva— se intenta aplicar su efecto de estado.
